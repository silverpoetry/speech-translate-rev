# Speech Translate Rev Web UI 设计指南

这份文档把实时转写页这轮原型定稿沉淀成可复用的设计规范，用于后续重构文件转写、模型选择、设置等选项卡。它不是泛泛的审美建议，而是一个可执行的界面契约：遇到布局争议时，以本文规则和真实 `1033x601` 截图为准。

## 1. 总体标准

界面目标是现代专业工具产品：克制、精密、安静、高效、有质感。它不应该像后台表单、卡片堆叠、AI 生成仪表盘、临时 demo 页面，也不应该靠花哨视觉掩盖信息架构问题。

核心原则：

- 用信息架构解决复杂度，而不是用更多卡片、标题和解释文案堆叠。
- 工具动作优先图标化；只有主动作或必须增强理解的动作使用“图标 + 极短文字”。
- 优先使用紧凑稳定的组件，不使用大面积假摘要卡。
- 状态直接显示，不写“这里会如何如何”的说明句。
- 页面只保留一个强主层级，避免到处都是局部小标题。
- 所有设计判断必须回到真实截图，而不是靠脑补。

明确禁止：

- 禁止廉价光效、玻璃拟态、大渐变、大阴影、强装饰。
- 禁止用删除有用功能来制造“简洁”。
- 禁止假控件、假摘要、假状态。
- 禁止静默 fallback 或兜底到旧坏方案。
- 禁止半截标题、小头、割裂卡片、错位滚动条。

## 2. 画布与截图契约

桌面 webview 的设计和验收必须使用真实窗口逻辑尺寸。

- 逻辑视口：`1033x601`。
- 左侧导航：约 `176px`。
- 主内容区：扣除侧栏和间距后的剩余宽度。
- 不优化极窄单列布局，除非程序最小宽度被正式修改。
- 截图必须使用和 app 一致的 viewport 与 scale factor。

页面是纵向工作台，由多个卡片块连续组成。首屏第一块不一定要撑满整个窗口。适当露出下一块是好的，但露出的内容必须像“可滚动工作台的下一段”，不能像错乱残片。

不合格的首屏延续：

- 只露出下一张卡片的上边缘。
- 只露出下一段标题栏。
- 底部出现半截标题或半个按钮。
- 卡片看起来互相重叠。

合格的首屏延续：

- 下一段至少露出完整标题和第一行有效控件。
- 滚动关系自然，能看出后面还有内容。
- 区块间距和页面节奏一致。

## 3. 基础设计 Token

以下数值是默认节奏。可以微调，但必须由真实截图证明更好。

间距：

- 页面外边距：`12px`。
- 区块间距：`12px`。
- 卡片内边距：`14px`。
- 控件间距：`8px`。
- 紧凑行内间距：`6px`。
- 双栏间距：`10px` 到 `12px`。

圆角：

- 页面容器：`16px`。
- 主 panel：`14px`。
- input / select / button：`10px`。
- chip / pill：`999px`。
- 小图标按钮：`9px` 到 `10px`。

高度：

- input / select：`34px`。
- icon button：`32px`。
- 主录制按钮：`36px`。
- chip：`24px` 到 `26px`。
- toggle 行：`28px`。
- 紧凑 toolbar：约 `46px` 到 `48px`。
- 输出区标题：避免独立厚标题栏，改用轻量内联标签。

字体层级：

- 页面标题：`15px / 700`。
- 区块标题：`13px / 700`。
- 字段 label：`12px / 650`。
- 正文：`13px / 500`。
- meta：`11.5px / 500`。
- chip 前缀：`10.5px` 到 `11px`，`650` 到 `700`。

颜色：

- 背景：`#f4f5f7`。
- 主 panel：`#ffffff`。
- 子表面：`#f8f9fb`。
- 次级表面：`#f2f4f7`。
- 边框：`#d9e0ea`。
- hairline：`#e8edf3`。
- 主文字：`#111827`。
- 次文字：`#536174`。
- 弱文字：`#7b8796`。
- 主操作：`#202b3a`。
- 主操作 hover：`#121a25`。
- 危险色：`#a33a3a`。

阴影：

- 只用低强度阴影，例如 `0 1px 2px rgba(...)`。
- 大 panel 可有非常轻的整体阴影。
- 不靠强阴影区分嵌套卡片。
- 如果阴影比内容更先被注意到，说明太重。

渐变：

- 渐变只能是非常弱的材质层次。
- 优先使用轻微纵向表面渐变。
- 禁止在内容区域使用明显斜向渐变或块状渐变。
- 禁止让输出区出现方形填充块。

推荐的输出区背景：

```css
background:
  linear-gradient(180deg, rgba(248, 250, 252, 0.42) 0%, rgba(255, 255, 255, 0) 56px),
  #fff;
```

不推荐的输出区背景：

```css
background:
  linear-gradient(145deg, rgba(244, 247, 251, 0.62), rgba(255, 255, 255, 0) 45%),
  #fff;
```

后者会形成突兀的方形或斜向色块，看起来像渲染错误。

## 4. 组件规范

### 4.1 Panel

Panel 是语义区块，不是装饰容器。

规则：

- 一个 panel 通常只承载一个概念单元。
- 不要卡片套卡片再套卡片。
- 不使用大摘要卡，除非它真是页面主信息。
- panel 内优先使用平整 field grid。
- 标题只命名功能，不解释工作流。

合格标题：

- `设备配置`
- `语言配置`
- `输出设置`
- `独立窗口`
- `模型检查与下载`

不合格标题或文案：

- `这里会持续追加结果`
- `右侧集中导入、导出与过滤`
- `只保留即时开关和调整`
- `先确定后端与模型，再统一加载到运行时`

### 4.2 Toolbar

主 toolbar 应该是一条精密控制行。

实时转写 toolbar 推荐结构：

- 左侧：主动作，`开始录制` / `停止录制`。
- 其后：转写和翻译开关。
- 中部：紧凑音量 meter。
- 右侧：必要的工具图标。

规则：

- toolbar 高度约 `46px` 到 `48px`。
- 音量 meter 不要做成大卡片。
- 录制相关控制不要散落到多张卡。
- toolbar 下方如果 chip 已能表达状态，不再放大状态卡。

### 4.3 Status Chip

chip 用于紧凑运行状态，不用于承载复杂信息。

合格例子：

- `空闲`
- `模型 small · faster-whisper`
- `语言 Japanese → Chinese`
- `录制 00:00:00`
- `缓冲 0/0 sec · 0 句`
- `输入 speaker`

规则：

- chip 必须单行、紧凑、可截断。
- chip 不能膨胀成卡片。
- 不显示 `等待检查` 这种可自动完成的假状态。
- 同一状态不要在多个地方重复展示。

### 4.4 图标按钮

工具动作优先使用图标按钮。

规则：

- 尺寸：`32px`。
- 圆角：`9px` 到 `10px`。
- 必须有 `title` 和 `aria-label`。
- 必须有 hover、focus-visible、active 状态。
- 主图标动作可以使用深色填充。
- 危险动作使用危险色或确认流程。

适合图标化的动作：

- 打开目录。
- 刷新。
- 保存。
- 打开独立窗口。
- 显示 / 隐藏 / 关闭独立窗口。
- 设置。

适合“图标 + 短文字”的动作：

- `开始录制`
- `停止录制`
- `一键导入并处理`
- `开始处理`

### 4.5 Toggle

toggle 用于布尔设置。

规则：

- toggle 行高：`28px`。
- switch 轨道要克制，不要立体或厚重。
- label 直接写状态名，例如 `置顶`、`无标题栏`、`点击穿透`。
- 不要给每个 toggle 写解释句。
- 不要为几乎相同的行为设置多个 toggle。

### 4.6 Field

字段要平整、对齐、可预测。

规则：

- input/select 高度：`34px`。
- label 在控件上方，不随机混排。
- 宽度允许时使用两列 grid。
- 长值必须省略截断，不能撑破布局。
- select 不能把相邻控件挤歪。

### 4.7 输出面板

输出面板是实时页的主视觉，但不能变成巨大空白。

规则：

- 转写 / 翻译左右等宽、等高。
- 标题使用轻量内联标签，不使用厚标题栏。
- 空态靠近顶部且低对比，不用巨大居中文案。
- 输出高度要和后续配置区平衡，不独占整屏。
- 可以使用非常轻的顶部纵向渐变，不允许可见方块填充。

不合格迹象：

- 每个输出面板有一条厚标题栏。
- TC/TL badge 像随机漂浮的小 pill。
- 空态文案巨大居中。
- 下一块内容完全不可见，整页像一个超大卡片。

### 4.8 独立窗口卡片

TC/TL 设置必须完整保留真实功能，同时保持紧凑。

每个 TC/TL 卡片包含：

- 标题和状态行。
- 状态驱动操作组。
- 尺寸和位置。
- 透明度。
- 字体。
- 字号。
- 字色。
- 背景色。
- 最大字符。
- 每行长度。
- 限制策略。
- toggle：置顶、无标题栏、点击穿透、置信度颜色、自动滚动、粗体、限制字符、限制每行。

操作状态：

- 未打开：显示打开和设置。
- 已打开且显示：显示关闭、隐藏、设置。
- 已打开但隐藏：显示关闭、显示、设置。

对齐规则：

- TC 和 TL 必须镜像。
- 顶线、标题基线、输入行、range 行、toggle 行必须对齐。
- 不能一边压缩、一边展开。

## 5. 各选项卡设计方向

### 5.1 实时转写

实时页是工作台，不是设置仪表盘。

推荐顺序：

1. 页面标题和独立窗口快捷入口。
2. 实时 toolbar。
3. 运行状态 chips。
4. 转写 / 翻译输出面板。
5. 设备和语言快速配置。
6. TC/TL 独立窗口设置。

移除：

- 大状态卡。
- 重复的模型加载入口。
- 解释性 helper 文案。
- 浮动且压住标题的 action row。
- 大音量面板。
- 重复说明 tab 含义的标题。

保留：

- 开始 / 停止录制。
- 转写 / 翻译开关。
- 实时音量 meter。
- 运行状态 chips。
- 输出面板。
- 设备刷新 / 保存。
- 语言保存。
- TC/TL 完整设置。

### 5.2 文件转写

文件页是队列和批处理工作台。

推荐结构：

1. 紧凑命令栏：导入、处理、转写开关、翻译开关。
2. 状态 chips：队列数量、处理状态、语言、引擎。
3. 队列表格 / 拖放区作为主对象。
4. 输出与过滤设置放在紧凑侧栏或下方 panel。
5. 高级语言和输出设置放在后续区块，不用摘要卡重复。

避免：

- 巨大的快捷操作卡。
- 重复的队列摘要卡。
- 比表格还大的空态卡。
- 多个自动打开目录 toggle。只保留一个 `任务完成后打开目录`。
- 输出格式按钮选中态不明确。

输出格式按钮规则：

- 选中态必须明显。
- 点击后必须立即更新本地状态，并能保存。
- 未选中态不能像 disabled。
- 使用统一 grid，不要大间距散落。

### 5.3 模型选择

模型页负责后端、缓存和下载。

推荐结构：

1. 运行配置：后端、模型、模型目录。
2. 模型检查与下载列表占满主要宽度。
3. 紧凑状态行：当前模型、引擎、缓存数量、下载状态。
4. 目录和缓存控制靠近模型列表。

避免：

- 假摘要状态卡。
- 在模型页展示窗口尺寸。
- 把设备策略放在模型页。
- 缓存显示 `等待检查`；能自动检查就自动检查。
- 在实时页 / 文件页重复模型加载设置。

### 5.4 设置

设置页是控制中心，但仍需要层级和节奏。

推荐结构：

1. 搜索和分类筛选。
2. 一个全局保存动作。
3. 按语义分组的设置区。
4. 如果需要，显示清晰的未保存状态。

避免：

- 顶部一堆保存按钮。
- 无动作价值的状态汇总卡。
- 搜索不筛选结果。
- 明显控件旁边写解释性废话。

## 6. 文案规则

界面文案要短、准、像产品，不像生成的说明书。

合格：

- `开始录制`
- `设备配置`
- `语言配置`
- `输出格式`
- `任务完成后打开目录`
- `模型检查与下载`

不合格：

- `右侧集中导入、导出与过滤`
- `这里会持续追加输出`
- `保留必要操作和当前配置，不再重复堆概览卡`
- `先确定后端与模型，再统一加载到运行时`
- `整理输入文件后统一启动处理`

判断规则：

- 如果一句话在解释“这个 UI 为什么存在”，删掉。
- 如果一个短语在命名“这个控件是什么”，保留。

## 7. 本轮实时页暴露出的反模式

### 7.1 半截小头

问题：

- 下一段只露出标题或上边缘。
- 看起来像组件被裁切或渲染错乱。

修法：

- 要么露出下一段完整标题和第一行有效控件。
- 要么完全放到折叠线以下。
- 不允许只露标题条。

### 7.2 输出区厚标题栏

问题：

- 每个输出面板顶部有一条 32px 左右的标题栏。
- TC/TL badge 像随机漂浮小组件。

修法：

- 使用轻量内联 label。
- 去掉厚标题条。
- 工具入口放到页面 header 或全局工具区。

### 7.3 方形渐变块

问题：

- 斜向渐变在输出区形成明显矩形色块。
- 空白区看起来像渲染 artifact。

修法：

- 只使用低透明度纵向渐变。
- 渐变控制在顶部很浅区域。
- 内容区不要用斜向大面积填充。

### 7.4 巨大空态

问题：

- 空态占据大半屏。
- 有用配置被挤到不可见，页面显得空且笨重。

修法：

- 空态提示靠近顶部。
- 输出区高度和后续配置区平衡。
- 让首屏体现多个工作区块的连续性。

### 7.5 卡片堆叠

问题：

- 卡片里套卡片，边框和标题重复。
- 页面噪音高，扫描困难。

修法：

- 一个语义区一个 panel。
- panel 内用 field grid。
- 只用少量 hairline 分隔。

### 7.6 假摘要卡

问题：

- 摘要卡重复控件信息。
- 显示 `等待检查`、`未加载` 这类没有动作价值的占位。

修法：

- 有用状态改为 chip。
- 设置项放回对应设置区。
- 能自动检查的状态自动检查。

## 8. 原型流程

正式改某个选项卡前，先做或更新独立原型。

原型规则：

- 使用独立 HTML/CSS 原型页。
- 原型未确认前，不改正式 `index.html`、`ui-preview.html`、`styles.css`、`app.js`。
- 原型 DOM 和 class 尽量使用最终可迁移结构。
- 不用图片假装 UI。

截图流程：

1. 使用 `1033x601` 真实 viewport。
2. 一次只截一个重点区域。
3. 对照本文验收标准逐项检查。
4. 在原型中修正。
5. 重复直到该区域通过。
6. 所有关键区域通过后，再迁移正式 UI。

复杂 tab 至少需要三张图：

- 首屏主工作区。
- 中段配置区。
- 下段高级设置或完整后段。

不要一股脑长截图后宣称检查完成。

## 9. 正式实现流程

原型确认后再落地正式 UI。

步骤：

1. 保留现有业务 ID 和 `data-action` 绑定。
2. 小 patch 迁移 DOM 结构。
3. 提炼或复用统一组件类。
4. 新样式生效后清掉旧重复 CSS。
5. 不留下互相覆盖的并行实现。
6. 删除已不需要的历史兼容路径，除非存在真实迁移需求。
7. 跑语法和单测。
8. 用正式 UI 截同样三张图。
9. 对比正式截图和确认过的原型截图。

工程约束：

- 可见控件必须可用。
- 设置控件必须能保存和恢复。
- API 失败必须走现有错误显示或至少 console error。
- 不允许静默失败。
- 不允许 fallback 到旧的坏路径。

## 10. 截图验收清单

每张截图都必须逐项通过。

- viewport 是 `1033x601`。
- 同级文字字号和字重一致。
- input/select/button 高度一致。
- 左右栏顶边和底边对齐。
- 镜像区标题基线对齐。
- 表单行跨列对齐。
- 空态克制，不支配页面。
- 没有大面积无意义空白。
- 没有重复入口。
- 没有重复标题。
- 没有解释性 AI 文案。
- 没有明显方形渐变块。
- 没有半截小头。
- 没有卡片堆叠噪音。
- 图标语义清楚。
- 图标按钮有 `title` 和 `aria-label`。
- 选中和未选中状态明确区分。
- disabled 不像坏控件。
- 可滚动页面能自然看出后续内容。

## 11. 工程检查

UI 结构或样式改动至少运行：

```powershell
git diff --check -- speech_translate\web\index.html speech_translate\web\ui-preview.html speech_translate\web\styles.css
```

正式 UI 改动按需运行：

```powershell
node --check speech_translate\web\app.js
python -m unittest test.web_bridge_api_test
python -m unittest test.detached_window_controller_test test.detached_windows_test
```

正式 UI 还要检查：

- `ui-preview.html` 和 `index.html` 结构保持一致。
- `app.js` 引用的 ID 仍存在。
- 所有 `data-action` 仍能派发。
- 所有设置输入仍能保存和恢复。
- 没有重复 CSS selector 互相覆盖布局。

## 12. 判断规则

拿不准时按这些规则决定：

- 如果元素只解释显而易见的事，删掉。
- 如果元素重复附近已有信息，改成 chip 或删掉。
- 如果控件有用但视觉太吵，重设计，不删除。
- 如果一个 panel 只有一两个小事实，它大概率不应该是 panel。
- 如果两个页面用不同方式解决同一种 UI，抽公共组件。
- 如果截图整体不错但某个细节像渲染 bug，先修细节再迁移。
- 如果正式 UI 和原型不一致，默认正式 UI 错，除非差异来自真实运行数据。

## 13. 当前实时页原型基准

当前定稿方向包括：

- 顶部紧凑 toolbar。
- 用运行 chips 代替状态卡。
- 转写 / 翻译输出面板左右等宽等高。
- 输出区无厚标题栏、无 TC/TL 角标小头。
- 输出背景只保留轻微纵向渐变，无方形色块。
- 设备和语言配置使用平整 field panel。
- TC/TL 独立窗口设置完整保留，并镜像对齐。
- 首屏底部露出有意义的下一段内容，而不是半截标题。

后续重构其他选项卡时，以这个基准作为视觉和交互质量下限，但信息架构要根据每个 tab 的真实任务重新设计。
